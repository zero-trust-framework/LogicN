# Galerina Trademark and Project Identity

This document explains how the **Galerina / Galerina** name, identity and branding should be used.

Galerina is intended to be released under the **Apache License 2.0**, which aLOws broad use, modification and redistribution of the project. However, the licence does not automatically allow people to use the Galerina name, branding or identity in a way that implies official approval.

---

## Project Name

The project name is:

```text
Galerina / Galerina
```

Short form:

```text
Galerina
```

Full form:

```text
Galerina
```

---

## Purpose of This File

This file exists to protect the project identity.

The goal is not to stop people using, forking or improving Galerina.

The goal is to prevent confusion between:

```text
the official Galerina project
forks of Galerina
modified versions of Galerina
commercial tools based on Galerina
unofficial compilers or runtimes
```

---

## Licence and Name Are Different

The Apache License 2.0 aLOws people to use, modify and distribute the project code and documentation under the licence terms.

However, the project name and branding should not be used to falsely imply official endorsement.

ALOwed:

```text
This project is based on Galerina / Galerina.
```

Not aLOwed without permission:

```text
Official Galerina Compiler
Official Galerina Runtime
Galerina Certified
Galerina Approved
Official Galerina Platform
```

unless the project is actually maintained or approved by the official Galerina maintainers.

---

## Acceptable Use

You may use the Galerina name to describe the origin of the project.

Examples:

```text
This tool is based on Galerina / Galerina.
This package supports the Galerina language.
This repository is a fork of Galerina.
This article discusses the Galerina language concept.
```

You may also use the Galerina name in technical compatibility statements.

Examples:

```text
Compatible with Galerina syntax draft 0.1.
Supports `.fungi` files.
Can parse galerina-style `boot.fungi` project files.
```

---

## Forks

Forks are aLOwed.

Forks should clearly state that they are forks.

Recommended wording:

```text
This project is a fork of Galerina / Galerina.
It is not the official Galerina project.
```

Forks should not present themselves as the official Galerina project unless approved by the maintainers.

---

## Modified Versions

Modified versions should clearly identify themselves.

Recommended wording:

```text
This project is based on Galerina / Galerina and includes modifications.
It is not the official Galerina project.
```

If a modified version changes language syntax, compiler behaviour, security rules or target outputs, that should be clearly documented.

---

## Unofficial Compilers

People may create their own galerina-compatible compilers or tools.

They should not call them official unless approved.

ALOwed examples:

```text
Experimental Galerina parser
Community Galerina formatter
Unofficial Galerina compiler prototype
galerina-compatible syntax highlighter
```

Avoid:

```text
Official Galerina compiler
Certified Galerina compiler
Approved Galerina toolchain
```

unless officially approved.

---

## Package Names

Packages should avoid names that imply official ownership unless they are part of the official project.

Preferred unofficial naming:

```text
community-galerina-parser
experimental-galerina-runtime
galerina-compatible-json-tools
```

Official package names may later use names such as:

```text
galerina-compiler
galerina-runtime
galerina-json
galerina-api
galerina-security
```

until package governance is defined, avoid implying official status.

---

## Logos and Visual Identity

If the Galerina project later creates a logo, icon, colour scheme or visual identity, those assets should not be used to imply official endorsement without permission.

ALOwed:

```text
showing the logo in an article about Galerina
using the logo to link to the official Galerina project
```

Not aLOwed without permission:

```text
using the logo on a commercial product to imply approval
using the logo on an unrelated language
using the logo on a fork that looks official
```

---

## Documentation Attribution

If you reuse Galerina documentation, preserve attribution.

Suggested attribution:

```text
Based on Galerina / Galerina documentation.
```

For modified documentation:

```text
This documentation is based on Galerina / Galerina and has been modified.
```

---

## Compatibility Claims

Compatibility claims should be accurate.

Good:

```text
Supports the Galerina 0.1 syntax draft.
Parses a subset of Galerina examples.
Experimental support for galerina-style API blocks.
```

Bad:

```text
Fully galerina-compatible
Official Galerina implementation
Production-ready Galerina compiler
```

unless those claims are true and approved.

---

However, commercial products should not imply official endorsement.

ALOwed:

```text
Our product includes code based on Galerina / Galerina.
```

Not aLOwed without permission:

```text
Official Galerina Enterprise Edition
Galerina Certified Cloud Runtime
Approved Galerina Commercial Compiler
```

---

## Project Identity Goals

The project identity rules are intended to:

```text
avoid user confusion
protect the original project reputation
allow open-source forks
allow commercial use
preserve attribution
encourage honest compatibility claims
```

They are not intended to block legitimate use of the code under Apache-2.0.

---

Forks can include this notice:

Use the Galerina name honestly.
