# Code of Conduct

## LogicN / LogicN Community Standards

LogicN / LogicN is an open-source language-design project for a strict, memory-safe, security-first, JSON-native, API-native and accelerator-aware programming language.

The goal of this project is to create a useful, respectful and technically focused environment where people can discuss language design, compiler design, security, APIs, JSON, AI-friendly tooling and future compute targets.

---

## Purpose

This Code of Conduct exists to make the LogicN project welcoming, constructive and safe for contributors.

The project should encourage:

```text
clear discussion
technical disagreement
respectful review
security-focused thinking
practical examples
open collaboration
careful language design
```

The project should discourage:

```text
hostility
personal attacks
gatekeeping
mocking beginners
unsafe advice
unsupported claims
spam
abuse
harassment
```

---

## Expected Behaviour

Participants are expected to:

```text
be respectful
be constructive
assume good intent where reasonable
give clear feedback
explain technical concerns
welcome new contributors
keep discussions relevant
respect project rules
respect licensing and attribution
```

Good feedback:

```text
This example weakens strict typing because it aLOws an implicit String to Int conversion.
Could we change it to use toInt() explicitly?
```

Poor feedback:

```text
This is stupid.
You clearly do not understand language design.
```

---

## Technical Disagreement

Technical disagreement is welcome.

LogicN is a language-design project, so disagreement is expected around:

```text
syntax
type rules
memory model
compiler architecture
security defaults
JSON handling
API design
webhook design
target outputs
AI tooling
```

Disagreement should focus on the idea, not the person.

Good:

```text
I think this syntax may be confusing because it mixes Option and Result behaviour.
```

Bad:

```text
Only an idiot would design it like this.
```

```text
safety
clarity
```

## Safety and Security Discussions

Security discussions should be handled carefully.

Participants should avoid:

```text
sharing real secrets
posting real API keys
posting production credentials
encouraging unsafe deployment
encouraging secret logging
encouraging insecure webhook handling
```

If a security concern is found, report it clearly and responsibly.

Examples:

```text
This example logs a SecureString.
This webhook example is missing replay protection.
This API route does not include a max body size.
```

---

## AI-Assisted Contributions

AI-assisted contributions are aLOwed, but contributors are responsible for reviewing them.

AI-assisted content should be checked for:

```text
accuracy
security
licence compatibility
project consistency
unsafe claims
real secrets
copied proprietary material
```

Do not submit AI-generated content blindly.

Unacceptable behaviour includes:

```text
harassment
abuse
threats
personal attacks
discriminatory language
sexualised language or imagery
trolling
spamming
doxxing
publishing private information
sharing credentials or secrets
deliberately misleading contributors
claiming false official endorsement
```

---

## No Gatekeeping

LogicN should be open to people with different experience levels.

Beginners, experienced developers, compiler engineers, security specialists, documentation writers and interested learners should all be treated respectfully.

Do not mock people for asking questions.

---

## No Unsupported Claims

Avoid making unsupported claims such as:

```text
LogicN is already production-ready.
LogicN makes all software secure.
LogicN can do impossible maths before a real backend exists.
LogicN has a real photonic backend before one exists.
LogicN prevents all reverse engineering.
```

Use accurate wording:

```text
LogicN is a concept.
LogicN is designed to support future photonic planning.
LogicN aims to make safer code easier to write.
```

---

## Enforcement

Project maintainers may respond to unacceptable behaviour by:

```text
giving a warning
editing or removing comments
closing issues or discussions
blocking contributors
restricting participation
reporting serious abuse to the platform
```

The response should be proportionate to the behaviour.

---

## Reporting Problems

If you see behaviour that violates this Code of Conduct, report it to the project maintainers.

A future project email or contact method should be added here:

```text
Maintainer contact: [ADD CONTACT METHOD]
```

When reporting, include:

```text
what happened
where it happened
who was involved
links or screenshots if available
why it is a concern
```

Do not publicly escalate sensitive reports unless necessary.

---

## Maintainer Responsibilities

Maintainers should:

```text
apply rules fairly
avoid favouritism
keep discussions constructive
protect contributors from abuse
encourage useful technical feedback
handle reports seriously
avoid public shaming
document major moderation decisions where appropriate
```
