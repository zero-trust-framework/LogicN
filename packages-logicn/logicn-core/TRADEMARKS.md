# LogicN Trademark and Project Identity

This document explains how the **LogicN / LogicN** name, identity and branding should be used.

LogicN is intended to be released under the **Apache License 2.0**, which aLOws broad use, modification and redistribution of the project. However, the licence does not automatically allow people to use the LogicN name, branding or identity in a way that implies official approval.

---

## Project Name

The project name is:

```text
LogicN / LogicN
```

Short form:

```text
LogicN
```

Full form:

```text
LogicN
```

---

## Purpose of This File

This file exists to protect the project identity.

The goal is not to stop people using, forking or improving LogicN.

The goal is to prevent confusion between:

```text
the official LogicN project
forks of LogicN
modified versions of LogicN
commercial tools based on LogicN
unofficial compilers or runtimes
```

---

## Licence and Name Are Different

The Apache License 2.0 aLOws people to use, modify and distribute the project code and documentation under the licence terms.

However, the project name and branding should not be used to falsely imply official endorsement.

ALOwed:

```text
This project is based on LogicN / LogicN.
```

Not aLOwed without permission:

```text
Official LogicN Compiler
Official LogicN Runtime
LogicN Certified
LogicN Approved
Official LogicN Platform
```

unless the project is actually maintained or approved by the official LogicN maintainers.

---

## Acceptable Use

You may use the LogicN name to describe the origin of the project.

Examples:

```text
This tool is based on LogicN / LogicN.
This package supports the LogicN language.
This repository is a fork of LogicN.
This article discusses the LogicN language concept.
```

You may also use the LogicN name in technical compatibility statements.

Examples:

```text
Compatible with LogicN syntax draft 0.1.
Supports `.lln` files.
Can parse logicn-style `boot.lln` project files.
```

---

## Forks

Forks are aLOwed.

Forks should clearly state that they are forks.

Recommended wording:

```text
This project is a fork of LogicN / LogicN.
It is not the official LogicN project.
```

Forks should not present themselves as the official LogicN project unless approved by the maintainers.

---

## Modified Versions

Modified versions should clearly identify themselves.

Recommended wording:

```text
This project is based on LogicN / LogicN and includes modifications.
It is not the official LogicN project.
```

If a modified version changes language syntax, compiler behaviour, security rules or target outputs, that should be clearly documented.

---

## Unofficial Compilers

People may create their own logicn-compatible compilers or tools.

They should not call them official unless approved.

ALOwed examples:

```text
Experimental LogicN parser
Community LogicN formatter
Unofficial LogicN compiler prototype
logicn-compatible syntax highlighter
```

Avoid:

```text
Official LogicN compiler
Certified LogicN compiler
Approved LogicN toolchain
```

unless officially approved.

---

## Package Names

Packages should avoid names that imply official ownership unless they are part of the official project.

Preferred unofficial naming:

```text
community-logicn-parser
experimental-logicn-runtime
logicn-compatible-json-tools
```

Official package names may later use names such as:

```text
logicn-compiler
logicn-runtime
logicn-json
logicn-api
logicn-security
```

until package governance is defined, avoid implying official status.

---

## Logos and Visual Identity

If the LogicN project later creates a logo, icon, colour scheme or visual identity, those assets should not be used to imply official endorsement without permission.

ALOwed:

```text
showing the logo in an article about LogicN
using the logo to link to the official LogicN project
```

Not aLOwed without permission:

```text
using the logo on a commercial product to imply approval
using the logo on an unrelated language
using the logo on a fork that looks official
```

---

## Documentation Attribution

If you reuse LogicN documentation, preserve attribution.

Suggested attribution:

```text
Based on LogicN / LogicN documentation.
```

For modified documentation:

```text
This documentation is based on LogicN / LogicN and has been modified.
```

---

## Compatibility Claims

Compatibility claims should be accurate.

Good:

```text
Supports the LogicN 0.1 syntax draft.
Parses a subset of LogicN examples.
Experimental support for logicn-style API blocks.
```

Bad:

```text
Fully logicn-compatible
Official LogicN implementation
Production-ready LogicN compiler
```

unless those claims are true and approved.

---

Commercial use of Apache-2.0 licensed LogicN code is aLOwed under the licence terms.

However, commercial products should not imply official endorsement.

ALOwed:

```text
Our product includes code based on LogicN / LogicN.
```

Not aLOwed without permission:

```text
Official LogicN Enterprise Edition
LogicN Certified Cloud Runtime
Approved LogicN Commercial Compiler
```

---

## Project Identity Goals

The project identity rules are intended to:

```text
avoid user confusion
protect the original project reputation
allow open-source forks
allow commercial use
preserve attribution
encourage honest compatibility claims
```

They are not intended to block legitimate use of the code under Apache-2.0.

---

Forks can include this notice:

```text

```


Use the LogicN name honestly.

You may say your work is based on LogicN, compatible with LogicN, or inspired by LogicN.

Do not imply that unofficial work is official, certified or endorsed unless that is true.
