### notes/51 — NumPy→TypeScript cheatsheet: does Galerina give full numeric/ndarray/linalg support?
- alreadyProcessed: partial
- verdict: refute
- summary: Full NumPy parity in core: NO, and by design. Galerina is governance-first, not a numeric kernel — the ndarray/broadcasting/linalg gap is deliberately delegated to the ext-bridge layer. Core ships governance OVER numeric shapes: `galerina-core-vector` validates Vector/Matrix/Tensor TYPES with zero arithmetic; `galerina-cpu-kernels` declares gemm/matmul/dot/ternary_matmul as governed PLAN descriptors. Real compute lives out-of-process behind `galerina-ext-bridge-cpp`/`-bitnet` (Microsoft BitNet.cpp ternary GEMM, MIT, AuditEvent CBOR Tag 410). At the .fungi level: 1D immutable lists + `for x in xs where c` filtered iteration (the np.where analog, shipped) + tree-walker floats; the bytecode VM rejects native floats (scaled-int ×1000); no mutable arrays, no 2D slice, no broadcasting, no eig/inv/det. None of the NumPy table is worth building into core. Dense linalg (inv/det/eig), if a real consumer appears, belongs behind a NEW Toxic-Border ext-bridge wrapping a vetted native lib — TRACK, do not build speculatively.
