## 1. The end-to-end SP 800-90 pipeline and fail-closed behavior

A NIST-conformant random-bit-generation stack is a three-layer pipeline, each layer specified by a distinct publication in the SP 800-90 series and each designed to **fail closed** (on any error condition, deny output rather than emit suspect bytes). A QRNG enters this stack as the *noise/entropy source* at the bottom; it is never a cipher primitive.

**Layer 1 — Physical noise source → entropy source (SP 800-90B, Jan 2018, [DOI 10.6028/NIST.SP.800-90B](https://doi.org/10.6028/NIST.SP.800-90B)).** Raw digitized samples from the (quantum/physical) noise source pass through mandatory continuous health tests *before any conditioning* (§4.3). Two approved continuous tests run (§4.4): the **Repetition Count Test** (§4.4.1), cutoff `C = 1 + ceil(-log2(alpha)/H)` — worked example in the standard: `H = 2.0, alpha = 2^-20 → C = 1 + 20/2.0 = 11` (§4.4.1) — which detects a "stuck" source; and the **Adaptive Proportion Test** (§4.4.2), window `W = 1024` for binary sources / `W = 512` for non-binary, with recommended false-positive `alpha` between `2^-20` and `2^-40` (§4.3) and `alpha = 2^-20` used in the worked examples (§4.4). On failure, "the entropy source **shall notify** the consuming application (e.g., the RBG) of the error condition... In the case where a **persistent failure** is detected, the entropy source **shall not produce any outputs**" (§4.3). Intermittent failures may be handled per a documented developer policy; the unconditional no-output rule attaches to *persistent* failure.

**Layer 2 — Entropy source → DRBG (SP 800-90A Rev. 1, Jun 2015, [DOI 10.6028/NIST.SP.800-90Ar1](https://doi.org/10.6028/NIST.SP.800-90Ar1)).** Approved mechanisms are Hash_DRBG, HMAC_DRBG, and CTR_DRBG (§10). The DRBG is seeded from the entropy source; when the internal `reseed_counter` exceeds `reseed_interval`, Generate returns a "reseed required" status and must reseed (or, with no reseed support, uninstantiate) before producing further output. On error, §11.4.2: "When a DRBG fails a health test or a catastrophic error is detected during normal operation, the DRBG **shall enter an error state** and output an error indicator. The DRBG **shall not perform any instantiate, generate or reseed operations** while in the error state, and pseudorandom bits **shall not be output** when an error state exists." Exiting the error state requires user intervention (e.g., power cycle) and re-instantiation.

**Layer 3 — DRBG + entropy source → RBG output (SP 800-90C final, 25 Sep 2025, [DOI 10.6028/NIST.SP.800-90C](https://doi.org/10.6028/NIST.SP.800-90C)).** SP 800-90C ties 90B + 90A into the RBG1/RBG2/RBG3/RBGC constructions (Section 4 below). Its failure handling is layered:
- An entropy source that detects a failure "returns a status indicating a failure. Other output is not provided"; a separate non-failure status ("cannot provide output at this time. Other output is not provided") also yields no output (§2.6 item 10).
- A failed physical entropy source "shall not be accessed to obtain entropy until the condition that caused the failure has been corrected and operational tests have been successfully passed"; "If all physical entropy sources [credited with providing entropy] report failures, the RBG operation **shall be terminated**" (§3.1 item 4, specifically 4.a.1 / 4.a.3; parallel summary in §8.1.2.1).
- Every RBG function returns a status that must be checked: "If the status code indicates the failure of an RBG component, then any other output returned **shall be considered invalid**" (§2.8), and RBG3 Generate returns a **Null bitstring** on error (§2.8.3.2).
- Entropy-accounting gate: "The Get_entropy_bitstring process shall not provide output for RBG operations unless the bitstring contains sufficient entropy to fulfill the entropy request" (§3.1 item 5).

**Net "unknown → deny" mapping.** Every layer exposes a status that gates output: a failed continuous health test takes the source out of service (no output until corrected and retested); the DRBG enters a no-output error state on a catastrophic error; the RBG terminates when entropy is unavailable and treats any non-success status as invalidating its output. A fail-closed governance interface should therefore treat *any* non-success status — or a requested security strength the DRBG cannot meet — as a hard deny (`rnd`/`keygen` returns no bytes).

## 2. SP 800-90B: the entropy-source contract

**NIST SP 800-90B, "Recommendation for the Entropy Sources Used for Random Bit Generation," final, January 2018** (Sönmez Turan, Barker, Kelsey, McKay of NIST; Baish, Boyle of NSA) specifies the design requirements and validation tests for entropy sources that feed the SP 800-90A DRBG mechanisms to build RBGs per SP 800-90C ([csrc.nist.gov/pubs/sp/800/90/b/final](https://csrc.nist.gov/pubs/sp/800/90/b/final); PDF [nvlpubs.nist.gov/.../nist.sp.800-90b.pdf](https://nvlpubs.nist.gov/nistpubs/SpecialPublications/nist.sp.800-90b.pdf)). The governing metric is **min-entropy** (worst-case), not Shannon entropy. The source is an entropy *source* sitting before any cipher — never a primitive. The Jan 2018 final remains the current standard (the "2pd" link is the superseded 2nd public draft, not a withdrawal).

**Estimation — two tracks.** A submitter first runs an IID test track; if the noise source passes, IID estimators apply. Otherwise the **non-IID track** runs the full estimator suite and the assessed entropy is the **minimum** over all estimators: most-common-value, collision, Markov, compression, t-tuple, LRS (longest-repeated-substring), plus four predictors — **MultiMCW, lag, MultiMMC, and LZ78Y** (SP 800-90B §6.3).

**Health tests — three categories:** start-up, continuous, and on-demand (§4.3–4.4). The start-up requirement is explicit: "The entropy source's startup tests **shall run the continuous health tests over at least 1024 consecutive samples**" (§4.3). Two continuous tests are approved:

- **Repetition Count Test (RCT):** detects a "stuck" source. Cutoff **C = 1 + ⌈−log₂(α)/H⌉**, where H is the assessed per-sample min-entropy and α the false-positive rate; recommended **α between 2⁻²⁰ and 2⁻⁴⁰** (§4.3), with 2⁻²⁰ a reasonable default (§4.4). Failure if any value repeats ≥ C times consecutively. Worked example: H = 2.0, α = 2⁻²⁰ → C = 11 (§4.4.1).
- **Adaptive Proportion Test (APT):** detects gradual entropy loss. Window **W = 1024 for binary** sources, **W = 512 for non-binary** (§4.4.2); count occurrences of the window's first sample; fail if the count ≥ cutoff C, where C is the **binomial critical value** for p = 2⁻ᴴ at α. There is no universal APT cutoff number — the cutoff is per-source (computed from each source's H and chosen α). Validated public-use documents show varied real values (e.g., C = 325 for one source; C = 69 for H = 4 bits per 8-bit sample at α = 2⁻²⁸), so cite the per-source public-use document rather than a fixed APT cutoff.

**Fail-closed:** on a **persistent** health-test failure the source "shall not produce any outputs" (§4.3); intermittent failures may be handled per documented developer policy.

**Conditioning (optional):** vetted components are HMAC (FIPS 198 with any FIPS 180/202 hash), CMAC (SP 800-38B with AES), CBC-MAC (Appendix F with AES), any approved hash, Hash_df, and Block_Cipher_df (from SP 800-90A) — **SP 800-90B §3.1.5.1.1** (not §6.3, which is the non-IID estimator section). Validation is administered through the NIST **Entropy Source Validation (ESV)** program under CMVP, with FIPS 140-3 IGs **D.J / D.K / D.O** defining the requirements for an entropy source in a FIPS-approved module. Per-source results are published as public-use documents on CSRC (e.g., E16, E100, E186) stating each source's assessed min-entropy and RCT/APT parameters.

## 3. SP 800-90A Rev. 1: approved DRBG mechanisms (condition + expand)

NIST SP 800-90A Rev. 1, *Recommendation for Random Number Generation Using Deterministic Random Bit Generators* (June 2015, [DOI 10.6028/NIST.SP.800-90Ar1](https://doi.org/10.6028/NIST.SP.800-90Ar1)), specifies **three** approved DRBG mechanisms: **Hash_DRBG**, **HMAC_DRBG**, and **CTR_DRBG** ([CSRC page](https://csrc.nist.gov/pubs/sp/800/90/a/r1/final); [full text PDF](https://nvlpubs.nist.gov/nistpubs/specialpublications/nist.sp.800-90ar1.pdf)). The headline change in Rev. 1 is the withdrawal of the fourth mechanism: "The Dual_EC_DRBG has been removed, and section numbers adjusted accordingly" (Appendix change-log).

**Condition + expand role.** A DRBG "uses an algorithm... that produces a sequence of bits from an initial value that is determined by a seed that is determined from the output of the randomness source" (§1). The seed is built by the *instantiate* function from entropy input + nonce + optional personalization string, and "depending on the DRBG mechanism and the randomness source, a derivation function **may be required** to derive a seed from the seed material" (§8.6.1) — that derivation function is the de-bias/decorrelation (conditioning) step. The deterministic *generate* function then expands that fixed-entropy seed into a large pseudorandom output stream, so a modest-rate noise source can back a high-rate output. This is bounded: output security strength = `min(output_length, DRBG_security_strength)` (§8.4). The five DRBG functions are instantiate, generate, uninstantiate (zeroize), and health test, plus an **optional** reseed (§8.1).

**Security strengths.** Only four are supported: **112, 128, 192, 256 bits** (enumerated in §8.4, "Security Strengths Supported by an Instantiation"; also §10.1 / Table 1). Per-mechanism: Hash_DRBG / HMAC_DRBG use SHA-1/224/256/384/512, with supported strengths referenced to SP 800-57 (Table 2). For CTR_DRBG the supported security strengths are likewise stated as "See [SP 800-57]" (Table 3 prints `keylen` = 168/128/192/256 and `outlen`, not the strengths). The corresponding strengths — 3-key TDEA → 112 (keylen 168 bits), AES-128 → 128, AES-192 → 192, AES-256 → 256 — derive from **SP 800-57**, not from explicit cells in Table 3. (Do not let TDEA's 168-bit keylen leak in as a security strength.)

**Reseed parameters (normative, from Tables 2 and 3).** `reseed_interval` is **2⁴⁸** for Hash_DRBG / HMAC_DRBG (Table 2) and for AES-based CTR_DRBG (Table 3), and **2³²** for 3-key TDEA CTR_DRBG (Table 3). `max_number_of_bits_per_request` is 2¹⁹ (Hash/HMAC, Table 2), `min(B, 2¹⁹)` for AES-CTR and `min(B, 2¹³)` for TDEA-CTR (Table 3). *(The earlier draft figure of "100,000 requests, Appendix B" was a fabrication — that value does not appear in SP 800-90A Rev. 1 and has been removed; the normative ceilings are the 2⁴⁸ / 2³² table values.)*

**Reseed & prediction resistance.** The (optional) reseed function "acquires new entropy input and combines it with the current internal state... to create a new seed and a new internal state" (§7.4). "Prediction resistance can be provided only by ensuring that a DRBG is effectively reseeded with fresh entropy between producing output for consecutive DRBG requests... an amount that is at least equal to the security strength" (§8.8) — i.e., prediction resistance requires fresh entropy ≥ security strength, available only when the randomness source has direct or indirect access to an entropy source or an NRBG.

## 4. SP 800-90C: RBG constructions (RBG1 / RBG2 / RBG3 / RBGC)

NIST SP 800-90C, *Recommendation for Random Bit Generator (RBG) Constructions* (final, **25 Sep 2025**, DOI [10.6028/NIST.SP.800-90C](https://doi.org/10.6028/NIST.SP.800-90C); PDF: [nvlpubs](https://nvlpubs.nist.gov/nistpubs/SpecialPublications/NIST.SP.800-90C.pdf); authors Barker, Kelsey, McKay, Roginsky [former NIST], Sönmez Turan) defines four construction classes, each pairing a 90A DRBG with a randomness source validated to SP 800-90B **or** SP 800-90C (a parent RBGC/RBG2/RBG3 can itself be the source) (§2.2). Each RBG targets indistinguishability from an ideal source: against an adversary doing 2^w work against an s-bit RBG (s ≥ w), advantage is ≤ 1/2 + 2^(w−s−1) + ε, with output limited to 2⁶⁴ bits and **ε ≤ 2⁻³²** (§2.1). "Full entropy" is defined as entropy/bit ≥ 1 − ε, ε ≤ 2⁻³² (§2.1).

**RBG1** (Sec. 4): no internal entropy source; seeded **once in its lifetime** over a physically secure channel from an external RBG2(P)/RBG3/root-RBGC. No reseed, **no prediction resistance, no full entropy** (Table 1). Strength is fixed at instantiation.

**RBG2** (Sec. 5): internal entropy source available **on demand**; optional reseed gives prediction resistance when performed. Variants RBG2(P) = physical, RBG2(NP) = non-physical. **Cannot provide full entropy**; output is at 128/192/256-bit security strength (Table 1).

**RBG3** (Sec. 6): one or more **physical** entropy sources, **continuously accessed**, producing **full-entropy** output with **prediction resistance on all outputs**; security strength equals requested output length. Only entropy from validated physical sources is counted (Method 1 of §2.3, invoked by §6.3 req 1: "Only the entropy provided by validated physical entropy sources shall be counted toward fulfilling entropy requests"). The DRBG is instantiated at its **highest security strength** (Table 3: AES-256 / SHA-2 / SHA-3 → 256), which is the **fallback strength if the entropy source fails undetected** — output then degrades gracefully to that RBG2(P)-level strength rather than failing (§6.1). Two sub-constructions:
- **RBG3(XOR)** (Sec. 6.4): n requested bits = n DRBG output bits XORed with n full-entropy bits from the validated physical source(s) (externally conditioned to full entropy if needed); the same entropy-source output must not be reused for both XOR and DRBG seeding. The DRBG **shall be reseeded before generating output if it has produced ≥ 2¹⁷ bits** since instantiation or the last reseed (§6.4.2 item 6). Full entropy holds as long as *either* the entropy source *or* the DRBG is sound.
- **RBG3(RS)** (Sec. 6.5): continuous reseeding — for each s-bit full-entropy output, **s + 64 bits of fresh entropy** are reseeded into the DRBG state before generating the s-bit string (§6.5.1.2), where s is the highest security strength of the DRBG primitive.

(For instantiation, §6.3 reqs 5–6: a CTR_DRBG without a derivation function needs keylen + output_len full-entropy bits; CTR_DRBG with df / Hash_DRBG / HMAC_DRBG need ≥ 3s/2 bits to instantiate at strength s and ≥ s bits per reseed.)

**RBGC** (Sec. 7): a tree of RBGC nodes on one computing platform; the root seeds/reseeds from an external RBG2/RBG3/full-entropy source, and non-root nodes seed from a parent RBGC. Prediction resistance is possible only at the root; non-root nodes provide neither prediction resistance nor full entropy (Table 1).

**Recommendation for a QRNG-backed design: target RBG3, specifically RBG3(XOR).** A quantum source is a *physical* entropy source — the only kind 90C counts for RBG3 (§6.3) — and RBG3 is the only class delivering full-entropy, prediction-resistant output. RBG3(XOR) keeps the output secure if *either* the QRNG *or* the DRBG is sound, while degrading gracefully to 256-bit DRBG strength under undetected source failure (fail-safe, consistent with the fail-closed posture); RBG3(RS) is the alternative when XOR-side full-entropy conditioning is impractical. **Note (honest framing):** SP 800-90C presents RBG3(XOR) and RBG3(RS) as *equally valid* RBG3 constructions; the preference for XOR here is an engineering judgment derived from the standard's stated guarantees, not an explicit NIST ranking. No single verbatim 90C sentence states the "secure if either is good" property in those words — it follows from the XOR-side full-entropy being DRBG-independent while the DRBG side survives undetected source failure at fallback strength.

## 5. The NIST ESV validation path (ESV vs CAVP vs CMVP)

For a QRNG to count as a *validated* entropy source under NIST, "validated" has a precise meaning: conformance to **NIST SP 800-90B (January 2018)** ([csrc.nist.gov/pubs/sp/800/90/b/final](https://csrc.nist.gov/pubs/sp/800/90/b/final)), demonstrated through the **Entropy Source Validation (ESV)** program, which the CMVP introduced for new FIPS 140-3 validations in 2022 (more specifically April 2022).

**What ESV requires.** The ESV Server "runs the SP 800-90B Entropy Assessment Tool on outputs provided by the entropy source to provide an entropy estimate. Additional documentation is needed to address the other requirements of SP 800-90B" ([ESV program page](https://csrc.nist.gov/Projects/cryptographic-module-validation-program/entropy-validations/esv)). SP 800-90B splits into an **IID track** and a **non-IID track**: statistical tests check whether outputs are independent and identically distributed; non-IID sources run roughly ten min-entropy estimators and the *minimum* estimate is taken conservatively ([SP 800-90B PDF](https://nvlpubs.nist.gov/nistpubs/SpecialPublications/nist.sp.800-90b.pdf)). Submission is via the **ESV Server** Web API: a **demo** server (available after 1/28/21, for client familiarization, cannot produce a certificate) and a **production** server "available for testing labs accredited to the **17ESV** scope," with the protocol derived from the ACVP/ACVTS family and authenticated by mTLS + TOTP ([ESV program page](https://csrc.nist.gov/Projects/cryptographic-module-validation-program/entropy-validations/esv); protocol detail in [usnistgov/ESV-Server](https://github.com/usnistgov/ESV-Server/blob/main/Entropy%20Source%20Validation%20Protocol.md), e.g. `POST /esv/v1/entropyAssessments`, "exactly one million samples, padded to one byte per sample"). Registration captures the noise source + conditioning components; data files (one million samples each) and PDF documentation justifying the non-testable SP 800-90B requirements are uploaded. A **Public Use Document (Template v1.2)** is "not required, only recommended" for standalone entropy validations ([Entropy Validation Documents](https://csrc.nist.gov/projects/cryptographic-module-validation-program/entropy-validations/documents)); the Entropy Assessment Tool is published at [github.com/usnistgov/SP800-90B_EntropyAssessment](https://github.com/usnistgov/SP800-90B_EntropyAssessment).

**Three distinct programs — do not conflate.** **ESV** validates an *entropy source* (SP 800-90B); **CAVP** validates *algorithms*; **CMVP** validates *modules* under FIPS 140-3. An entropy source is a *component* of a validated module — SP 800-90B with FIPS 140-3 IGs **D.J, D.K, D.O** governs its inclusion. Accredited labs submit conformance justifications to the CMVP to receive an **Entropy Validation Certificate** (entropy source, per 90B) or a **Random Bit Generator Validation Certificate** (RBG, per 90C). SP 800-90A Rev. 1 (June 2015) supplies the DRBG; **SP 800-90C is now final (25 Sep 2025, DOI [10.6028/NIST.SP.800-90C](https://doi.org/10.6028/NIST.SP.800-90C))** and defines the RBG constructions ([SP 800-90C CSRC](https://csrc.nist.gov/pubs/sp/800/90/c/final)).

**Caveat / "ENT" flag.** On a module certificate, **ESV** denotes a 90B-validated source under the new program, whereas older **ENT (P) / ENT (NP)** caveats mark legacy/passively-received entropy without a standalone ESV cert. This is unrelated to the Fourmilab `ent` command-line tool — calling `ent` (or dieharder) output "ENT validation" is a misnomer; neither tool constitutes NIST validation.

## 7. FIPS-204 (ML-DSA) entropy binding and the binding rules

**KeyGen requires an approved-RBG seed at parameter-set strength.** FIPS-204 `ML-DSA.KeyGen` (Algorithm 1, described in §5.1) "uses an RBG to generate the 256-bit random seed ξ. The seed ξ **shall** be a fresh (i.e., not previously used) random value generated using an approved RBG, as prescribed in SP 800-90A, SP 800-90B, and SP 800-90C" (§3.6.1, [FIPS-204](https://nvlpubs.nist.gov/nistpubs/fips/nist.fips.204.pdf)). The RBG "**shall** have a security strength of at least 192 bits for ML-DSA-65 and 256 bits for ML-DSA-87. For ML-DSA-44, the RBG **should** have a security strength of at least 192 bits and **shall** have a security strength of at least 128 bits. If an approved RBG with at least 128 bits of security but less than 192 bits of security is used, then the claimed security strength of ML-DSA-44 is reduced from category 2 to category 1" (§3.6.1). This is the SP 800-90 pipeline (90B entropy source → 90A DRBG → 90C construction) feeding ML-DSA: the QRNG validates as the entropy source feeding an approved DRBG, and the DRBG output is the seed.

**Hedged Sign consumes a 256-bit `rnd` and degrades gracefully.** In the default hedged variant (Algorithm 2, line 5; described in §5.2, with the concept in §3.4), the signer "uses an approved RBG to generate a 256-bit (32-byte) random seed `rnd`. If the deterministic variant is desired, then `rnd` is set to the fixed zero string {0}³²." The Connolly CFRG draft restates this: "The hedged variant degrades gracefully: if the random number generator provides no entropy (all zeros), the hedged variant produces the same output as the deterministic variant" ([draft-connolly-cfrg-ml-dsa-security-considerations-02](https://www.ietf.org/archive/id/draft-connolly-cfrg-ml-dsa-security-considerations-02.html), §2.2.1). The benefit is hardening, not strength: "The primary purpose of `rnd` is to facilitate countermeasures to side-channel attacks and fault attacks on deterministic signatures... even a weak RBG may be preferable to the fully deterministic variant" (FIPS-204 §3.6.1). Note an asymmetry vs. the KeyGen seed: `rnd` "should ideally be generated by an approved RBG, [but] other methods for generating fresh random values may be used" (§3.6.1) — a stricter fail-closed library may nonetheless require an approved RBG for `rnd` as well (a policy choice, not a FIPS-204 requirement).

### Binding rules (fail-closed)

1. **A QRNG is a SOURCE, not a primitive.** It feeds the SP 800-90 chain (90B noise source → 90A DRBG → 90C construction); ITU-T [X.1702 (approved 13 Nov 2019)](https://www.itu.int/dms_pubrec/itu-t/rec/x/T-REC-X.1702-201911-I!!SUM-HTM-E.htm) standardizes exactly this: it "defines a generic functional architecture of a quantum entropy source, a common method to estimate and validate the entropy of a noise source under evaluation, and a common method to specify randomness extractors."
2. **Raw QRNG bits → key is PROHIBITED.** FIPS-204 mandates ξ come from an *approved* RBG (90A/B/C), not raw noise. Raw bits are biased/correlated; they must pass conditioning and SP 800-90B health tests first.
3. **The signing benefit is side-channel/fault hardening, NOT more unforgeability.** Unforgeability comes from the lattice problem — FIPS-204 §3.2 states "The security of ML-DSA is based on the MLWE problem... and a nonstandard variant of MSIS called SelfTargetMSIS," and §3.1 designs ML-DSA to be strongly existentially unforgeable under chosen-message attack (**SUF-CMA**); the Connolly draft frames this as EUF-CMA (§3.2). Either way the property is independent of `rnd`. ML-DSA-44/65/87 are claimed in security strength categories 2/3/5 respectively (FIPS-204 §4, Parameter Sets). A perfect QRNG does not raise the category; a dead QRNG (all-zeros `rnd`) still yields a valid, unforgeable signature via the deterministic path — it merely loses fault resistance (and the deterministic variant "is vulnerable to fault injection attacks," Connolly §3.2.1).
