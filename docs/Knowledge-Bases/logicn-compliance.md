## The Core Shift

Traditional compliance asks:

```
Did we follow the rules?
Can we prove it?
What happened?
Who authorised it?
```

These are answered **after the fact** — by parsing logs, interviewing engineers, reconstructing event sequences, and hoping the audit trail is complete.

LogicN answers them **before execution**:

```
What is allowed?                  → GovernanceGraph
Why is it allowed?                → ProofGraph
Who authorised it?                → CapabilityGraph
What data did it touch?           → PrivacyGraph + LineageGraph
What did it cost?                 → CostGraph
What happened?                    → AuditGraph
```

The compliance record is not assembled from logs.
It is **generated by the execution architecture**.

## How It Works

### What Governance Provides

```
GovernanceGraph  → what is allowed, by whom, under what conditions
ProofGraph       → proof of legality before execution
CapabilityGraph  → who holds which capabilities
PrivacyGraph     → what protected data exists and how it must be handled
LineageGraph     → where data came from and where it went
AuditGraph       → immutable record of every governed execution
CostGraph        → what every execution cost
```

### The Execution Flow

```
[ Traditional Software ]
Deploy code → run → breach/error → investigate → reconstruct → fine

[ LogicN ]
Compile → ProofGraph rejects illegal flows → governance proven before execution
```

If a developer writes code that routes unredacted patient data to an unauthorised external model, **the compilation fails**. The breach never happens. There is no incident to investigate.

## Regulatory Compliance Mapping

### EU AI Act — Article 12 (Event Logging for High-Risk Systems)

**Requirement**: High-risk AI systems must maintain untampered, automated event logging across their lifecycle.

**LogicN response**:
- `AuditGraph` is append-only and immutable — cannot be altered post-execution
- `InputSeal` + `OutputSeal` for any NPU/TPU/ANE target — mathematically proves what entered the AI model and what came out
- `ProofGraph` records which policies were applied before execution
- `logicn-verify proof.json` CLI: any auditor can offline-verify the proof chain

**Gap closed**: EU AI Act requires logging for *black-box* models. LogicN's Input/Output Seal means the internal state of an NPU or TPU never needs to be inspected — the seals prove the transformation boundary cryptographically.

---

### EU AI Act — Article 13 (Transparency for Users)

**Requirement**: High-risk AI systems must be transparent about their capabilities and limitations.

**LogicN response**:
- `contract.ai { approved_models { ... } }` — the permitted models are declared in source code, not a configuration file
- `AuditGraph` records which model was invoked and with what inputs
- `GovernanceSignature` (Phase 39) provides unforgeable evidence the ProofGraph was issued by a specific compiler version

---

### HIPAA — Security Rule (§164.312)

**Requirement**: Access controls, audit controls, integrity, and transmission security for PHI.

**LogicN response**:
- `contract.privacy { pii { ... } require protected_boundary before ai.infer }` — enforced at compile time
- `PrivacyGraph` tracks PII through every flow
- `LineageGraph.retention 7_years` — automatic data retention compliance
- Audit controls: `AuditGraph` records every PHI access with actor, trace_id, timestamp
- Integrity: `InputSeal`/`OutputSeal` prove data was not modified in transit to accelerators
- `require local_execution` in AI contracts — PHI never leaves the device

### SOC 2 — Trust Service Criteria

**Requirement**: Security, availability, processing integrity, confidentiality, privacy.

| Criterion | LogicN Component |
|---|---|
| Security (CC6) | `CapabilityGraph` — no access without declared capability |
| Availability (A1) | `contract.limits { request_time 100ms }` — SLA declared in source |

| Confidentiality (C1) | `PrivacyGraph` + `require redaction before audit.write` |
| Privacy (P1-P8) | `LineageGraph` + `contract.privacy` — full GDPR/CCPA/HIPAA mapping |

### SEC Rule 17a-4 — Records Retention and Authenticity

**Requirement**: Financial records must be unalterable, authentic, and reproducible.

**LogicN response**:
- `AuditGraph` is append-only — records cannot be deleted or altered
- `GovernanceSignature` (Phase 39) provides unforgeable proof of who generated each compliance certificate
- `InputSeal`/`OutputSeal` on every financial model execution — auditors can verify any transformation
- `LineageGraph` — full data lineage for every financial calculation

### ISO 27001 — Information Security Management

**Requirement**: Systematic management of information security through controls.

**LogicN response**:
- All controls declared in source — `contract.effects`, `contract.privacy`, `contract.ai`
- Controls enforced at compile time — not configurable at runtime by operators
- `AuditGraph` provides continuous evidence of control effectiveness
- `CapabilityGraph` implements ISO 27001 Annex A.9 (Access Control) structurally
