# Automated Runtime Trust Strategy

## Definition

LogicN should make trust mostly automatic through compiler-generated metadata,
runtime provenance and automatic verification. Normal developers should not
manually write identity, attestation, route or provenance declarations.

```text
Developer writes simple secure code.
Compiler extracts trust requirements.
Runtime generates a Trust Capsule.
Build system signs the artifact.
Registry verifies packages.
Runtime verifies before execution.
Provenance is recorded automatically.
AI code is risk-scored automatically.
Humans only approve high-risk exceptions.
```

## The Trust Capsule

The **LogicN Runtime Trust Capsule** is generated by the compiler/runtime, not
hand-written by developers. It records:

```text
project identity
source hashes
flow hashes
package hashes
permission map
safe/unsafe boundary map
query map
API boundary map
runtime profile
risk score
build provenance
artifact signature
verification policy
```

## Trust Levels

### Level 0: Development Trust

```text
self-signed project root
automatic local provenance
local trust capsule
no manual identity required
no external transparency log required
```

Commands: `logicn init`, `logicn build`, `logicn run`

### Level 1: Team Trust

```text
project trust root
package lock verification
team registry trust
automatic provenance
CI-generated signatures
```

### Level 2: Production Trust

```logicn
runtime profile production {
  trust: automatic
  provenance: required
  signed_packages: required
  build_attestation: required
  privileged_flows: policy_gated
}
```

### Level 3: Enterprise Trust

```logicn
runtime profile enterprise {
  trust: strict
  workload_identity: required
  transparency_log: required
  registry_policy: strict
  route_policy: generated
  human_approval: high_risk_only
}
```

## AI-Generated Code Trust

Risk is classified automatically, not by manual approval every time.

Low-risk AI code (formatting, no privileged authority):

```text
risk: low
action: allow local/staging automatically
```

High-risk AI code (writes to financial data, external input):

```text
risk: high
action: block production until approved by policy

LNN-AI-PRIV-001:
AI-generated high-risk flow requires production approval.
```

## Self-Sign Policy

`self-sign` should not mean "trust anything because the project says so."

| Context | Sign Mode |
| --- | --- |
| Local development | Local self-signed (allowed) |
| Private/internal projects | Project self-signed (allowed) |
| Public packages/production | Registry or CI-anchored signing (required) |
| Regulated systems | Enterprise external trust (required) |

A production runtime rejects local signatures:

```text
LNN-TRUST-041:
Local self-signed artifact cannot run in production profile.
Use CI/OIDC, trusted registry, or organisation signing.
```

## Reference Standards

LogicN combines lessons from:

```text
Sigstore / Cosign  — keyless signing, OIDC identity, transparency logs
SLSA Provenance    — build trust generated by pipeline, not hand-written
SPIFFE / SPIRE     — workload identity issued to running services automatically
in-toto            — attestations generated as artifacts, not manual declarations
TUF                — package trust via signed registry metadata and roles
OpenSSF Scorecard  — automated risk scoring, human approval for exceptions only
```

## Three Architecture Layers

### 1. Core Language (all developers)

```text
flow, fn, safe, unsafe
validate, trust.data, encode.*
Query, GlobalVault, uses
```

Goal: secure everyday programming without heavy ceremony.

### 2. Automated Runtime Trust (all projects, mostly invisible)

```text
trust capsule
source/flow hashing
package verification
risk scoring
build/runtime provenance
automatic audit events
```

Goal: make secure behaviour the default.

### 3. Enterprise Governance (platform/security teams only)

```text
runtime identity
workload identity
transparency logs
strict route policy
manual approval gates
private registries
external trust roots
HSM/KMS integration
```

Goal: zero-trust runtime operation without burdening normal developers.

## Bootstrap Runtime Plan

LogicN cannot self-host immediately. The bootstrap runtime uses Node.js with
strict JavaScript contracts to validate the governance model:

```text
Phase 1: Node.js bootstrap runtime
Phase 2: Governed IR stabilisation
Phase 3: native components in Rust/WASM/C/LogicN
Phase 4: partial self-hosting
Phase 5: LogicN runtime signs and verifies its own governed artifacts
```

## CLI Design

```bash
logicn init         # create project trust root
logicn build        # build with trust capsule generation
logicn run          # verify trust capsule then execute
logicn trust inspect # show trust status
logicn build --profile production  # production trust enforcement
logicn trust verify --profile production
logicn deploy       # deploy after trust verification
```

## Core Principle

```text
LogicN identity should be real.
LogicN identity should be automatic.
LogicN identity should not depend on every developer
manually writing perfect identity data.
```
